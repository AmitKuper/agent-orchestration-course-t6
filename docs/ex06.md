## 3 Central Task Definition and Orchestration Challenge

The primary task is planning, implementation, and execution of **a complete end-to-end game pipeline**, which allows two AI agents — the autonomous **"Cop"** and **"Thief"** — to conduct a dynamic pursuit game on a 2D grid between them. The system requires both agents to:

1. **Decode** each other's messages in natural language;
2. **Draw conclusions** about the opponent's location from partial observation;
3. **Translate** insights into physical moves on the grid network.

**The central success metric** is the communication and orchestration ability of the agent pair — **not** the game strategy algorithm. Every group is required to present a valid series of games between a pair of agents (Cop and Thief) belonging to the same group, managed on its own MCP servers in an **autonomous manner — without human intervention** — until sending the organized summary report.

## 4 Game Rules, Grid, and Scoring System

### 4.1 Structure of Sub-Games and the Game

The process is based on a clear distinction between two levels:

- **Sub-game** (Sub-game): A single pursuit round, limited to at most 25 moves (Moves). In each move, players are allowed to change position or perform an action. Usually the Thief moves first, followed by the Cop, and so on cyclically.
- **Game**: A complete series of 6 consecutive sub-games; the results of all sub-games are accumulated and reported together at the end of the series.

### 4.2 Game Board

The game board is a 2D grid with a default size of 5×5 squares. The system is required to support **flexible architecture** that allows dynamic resizing of the board through a configuration file (not hard-coded). Each square has Cartesian coordinates, and the Cop and Thief start at positions on the board (random or chosen as a strategy). Movement is allowed in all directions, including diagonally. The game is essentially a **state machine**: the board state changes with each move.

### 4.3 Victory Conditions and Setting Barriers

- **Cop victory**: Achieved when the Cop arrives at exactly the same square where the Thief is located (capture).
- **Thief victory**: Achieved if the Thief survives 25 moves without the Cop landing on his square throughout the entire sub-game.
- **Setting Barriers** (Barriers): As an alternative action to movement, the Cop may place a barrier. This action **blocks the current square where he stands** and prevents advancement from it. However, the blocked square becomes **impassable for the Thief** (similar to a wall or edge of the board) while the Cop himself counts as **one**. The Cop is limited to a maximum of **5 barriers** per sub-game; the Thief cannot place barriers.

### 4.4 Scoring System

The score is calculated separately for each sub-game:

**Table 1: Score for Individual Sub-game**

| Sub-game Outcome | Cop Score | Thief Score |
|---|---|---|
| Cop wins (Cop wins) | 20 | 5 |
| Thief wins (Thief wins) | 5 | 10 |

The maximum possible score for a group in the full game is 90 points (3×20 as Cop plus 3×10 as Thief), and the minimum score is 30 points.

### 4.5 Gradual Sanity Checks

To perform integration checks and prevent technological failures at advanced stages, it is recommended to perform Sanity Checks at gradual board dimensions:

**Table 2: Recommended Sanity Check Stages**

| Stage | Grid Dimensions | Validation Goal | Complexity |
|---|---|---|---|
| 1 | 2x2 | Algorithmic sanity check, basic Pipeline integration and message transfer verification | Very Low |
| 2 | 3x2 / 2x3 | Examining coordination mechanisms, hyper-parameter tuning and failure identification | Medium |
| 3 | 4x3 / 3x4 | Examining the effect of observation ambiguity; initial distance over vision radius | High |
| 4 | 5x5 | Final test run, graph production, full game results analysis | Maximum |

## 5 MCP Server Architecture and Natural Language Communication

### 5.1 Two Servers, Free Language, and Tools

MCP (Model Context Protocol) is an open standard for connecting language models (LLMs) to tools, data sources, and external services. **The base of the orchestration is two independent and self-contained MCP servers**: one for the Cop and one for the Thief. The agents do **not** communicate via a rigid protocol transmitting numerical positions — but in **free natural language**. Each agent generates textual messages describing its intentions, its local observations, or deception attempts; the receiving agent uses **Tools** (exposed via the MCP server) to read the message, update the state, and choose its next action. Each agent's MCP server is built using the **FastMCP** package, which exposes tools to the external world for mutual verification, sending and receiving messages, and location packages.

### 5.2 MCP Client vs MCP Server

An important architectural point: the language model (LLM) **is not the server or the operator**. **Within the MCP server**, the server is a separate component exposing tools, data sources, and prompt architectures (Prompts). **The client** (MCP Client) — which is the game engine or the **orchestrator** — is what the students build. It manages the duel dialogue and the logic. Workflow: the client sends a query to the LLM, receives back a decision to activate tools (Tool Call), calls the MCP server to execute the action, and returns the result to the LLM to complete the loop.

## 6 Deploying MCP Servers and Managing Them

- **Stage One — Local Run**: Setting up and running both servers on the local computer (Localhost) on **separate ports**, verifying that the game engine and agents recognize each other and maintain full interaction, playing against each other.
- **Stage Two — Cloud Deployment**: After verifying local Pipeline correctness, uploading the MCP servers to a public cloud (e.g., Prefect Cloud or a similar distributed platform).
- **Security and Authentication**: It is mandatory to embed an authentication mechanism based on **Token** (Authentication Token) that enables blocking and revoking access (Revoke) when needed, to prevent non-authorized third-party access.

**Security and Network Critical Emphasis**: It must be verified that the MCP server URLs are accessible entirely from the public internet and are not blocked by a firewall. Therefore, it is **not recommended** to run or test the servers from within the workplace or university network, which tend to block outgoing/incoming traffic on non-standard ports. Each group needs **two** URLs — one for the Cop and one for the Thief.

## 7 LLM Architecture: Three Access Approaches to Connect the Language Model

Even in the cloud, the MCP server still needs an LLM. Three architectural approaches stand:

### 7.1 Approach 1: Cloud Model Public API (Simple and Recommended)

Instead of running a local model and dealing with its exposure, the MCP client (your code) connects with an API key to a public cloud provider's service (such as OpenAI, Anthropic, or Gemini). **Operation mechanism**: the game engine sends an HTTP request with the API key directly to a public cloud provider, the API returns the LLM response (containing the Tool Call decision), and routes the call to the MCP server. **Advantages**: stable and fast solution, no need to expose the personal computer or worry about firewall issues. After the short conversations and low token consumption, you can even use basic or nearly free-cost models.

### 7.2 Approach 2: Secure Exposure of Local Ollama to the Cloud

Ollama runs local models and uses by default port 11434 on the Loopback (127.0.0.1); its direct exposure without a built-in security mechanism is a severe cybersecurity risk. Therefore, if you want to run a local LLM while the MCP server is in the cloud, you must use **tunneling** (Tunneling): a layer that adds security to the MCP server in the cloud:

- **ngrok with traffic security policies** (Traffic Policy): Creates a public HTTPS URL that redirects to local Ollama, with access restriction using Basic Auth (configured in the ollama.yaml file), running ngrok's CLI pointing to the port and policy file. The MCP server receives requests only with an authentication header (Authorization Header).
- **Localtonet**: Similar to ngrok, opens a public tunnel to 127.0.0.1:11434, with a convenient option to enable HTTP Authentication through the management dashboard.
- **Reverse Proxy Server Nginx** (for full engineering implementation): Ng-inx server performs SSL Termination, serves as Reverse Proxy for local Ollama, requires authentication via htpasswd file (using Certbot / Let's Encrypt), and blocks Ollama's port in the local firewall (UFW / nftables).

### 7.3 Approach 3: Hybrid Architecture (Recommended for Secure Local Development)

Instead of exposing local Ollama, the language model and the client (the orchestrator/game engine) remain on the local computer, and only the MCP servers are saved in the cloud; the game client and the Cop and Thief run locally via Ollama (local LLM); the game client connects to Ollama at localhost:11434 **without any exposure at all**. The client needs to connect to the MCP server via HTTPS outbound (Outbound) to the cloud servers. **Why is this more secure**: the outgoing request does not require opening incoming ports (Inbound) or exposing an IP; Ollama stays behind the local firewall, and only MCP servers (which do not contain keys or sensitive information) are publicly accessible.

## 8 Simple Q-Table Reinforcement Learning Algorithm: Recommendation

Use of reinforcement learning (Reinforcement Learning) is **optional and recommended only** for groups genuinely interested in developing decision-making mechanisms. Groups not based on heuristics, distances, or LLM model guidance (Prompt Engineering) are permitted. Dr. Segal notes that Q-Learning presents an adaptive winning game strategy and proposes Tabular Q-Learning as basic learning without the need for deep neural network training.

### 8.1 State, Action, and Value

Three foundational building blocks: **state** (s, State) — where the agent is located; **action** (a, Action) — the move (movement in any direction); **reward** (r, Reward) — the quality score of the state. Dr. Segal computes with a treasure-seeking agent example: each step costs a small negative reward (small negative), falling into a pit incurs a large penalty, and reaching the treasure earns a positive reward; through trial and error across repeated episodes, the agent learns which states are more valuable.

### 8.2 Q-Table and Bellman Equation

In the Q table, each row represents a state s, each column represents an action a, and the value Q(s, a) represents the "quality" of the action — the expected accumulated long-term reward. In each step the agent chooses an action according to an Epsilon-Greedy exploration/exploitation policy, receives an immediate reward r, and moves to the next state s'. The table update is performed using the Bellman Equation:

$$Q(s, a) \leftarrow Q(s, a) + \alpha \left[ r + \gamma \max_{a'} Q(s', a') - Q(s, a) \right]$$

Where α is the **learning rate** (Learning Rate), with common values between 0.01 and 0.5, determining how much the new information influences the existing value; γ is the **discount factor** (Discount Factor), between 0 and 1, representing the importance of future rewards; and max_{a'} Q(s', a') is the maximum expected reward from the next state. Below is minimal code that computes the Q-table update according to the Bellman equation:

#### Q-Table Update According to Bellman Equation

```python
import numpy as np

# 5x5 grid -> 25 states, 4 movement actions
num_states = 25
num_actions = 4
q_table = np.zeros((num_states, num_actions))

# learning hyper-parameters
learning_rate = 0.1
discount_factor = 0.9

def update_q_table(state, action, reward, next_state, done):
    # max Q for the next state
    best_next_q = 0.0 if done else np.max(q_table[next_state])
    # TD target and TD error
    td_target = reward + discount_factor * best_next_q
    td_error = td_target - q_table[state, action]
    # Q-value update
    q_table[state, action] += learning_rate * td_error
```

This method allows agents to improve the action policy over time through interaction, trial, and error, without requiring heavy computational resources or deep learning libraries.

## 11 Submission Requirements and Scientific README

The submission includes two main deliverables: **source code uploaded to a GitHub repository** containing all source code, and a scientific analytical documentation file named README.md at the repository root. The report must be written in high-level scientific language and include the following:

- **Formal Modeling**: Presenting the pursuit game as a partially observable Dec-POMDP (Decentralized Partially Observable Markov Decision Process), defining the appropriate mathematical tuple, detailed state space, and the observations. The general tuple is:

$$\langle n, S, \{A_i\}, P, R, \{\Omega_i\}, O, \gamma \rangle$$

Where n is the number of agents, S is the state space (positions of players and barriers), {A_i} are the actions of each agent (movement and barrier placement), P is the transition function, R is the reward function (scoring table), O is the space of partial observations, {Ω_i} is the observation function, and γ is the discount factor.

- **Orchestration Challenge Analysis**: An in-depth discussion of the difficulties in managing free natural language communication without a pre-defined protocol, ways of coping with linguistic ambiguity, and methods for ensuring mutual understanding.

- **Visualization and Learning Proof** (if done): Graphs of learning curves (using Q-Table), display of the game board's graphical dashboard (GUI), and logs from the CLI and from the MCP servers in the cloud indicating valid communication.

## 14 Key Insights

- **Infrastructure before the game**: The task value is the ability to orchestrate — setting up two autonomous AI agents that match protocol in natural language via MCP servers — not in the game outcome or strategy.
- **Free language, no rigid protocol**: Agents are autonomous and independent; each understands the other, the internal implementation does not matter. The problem is formally modeled as Dec-POMDP with partial observation.
- **Server/Client separation**: The language model lives in the client (the orchestrator), while the MCP server only exposes tools.
- **From local to cloud**: The progression starts at localhost, separate ports — to cloud deployment — through dealing with domains, tokens, tunneling and cybersecurity.
- **Three LLM approaches**: API cloud, Ollama local exposed via tunnel, or hybrid architecture with only Outbound requests.
- **Security and automation**: Tokens (OAuth, ngrok Traffic Policy) replace passwords, and Gmail API enables autonomous JSON reporting.
- **Soft skills**: Team work, management under uncertainty and time pressure — inspired by the Prisoner's Dilemma.
